**⚠️ 实战视角的修正与补充建议（强烈建议修改）：**

| **审查维度**       | **潜在坑点**                                                 | **优化建议 (Best Practice)**                                 |
| ------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **Docstring 流派** | 指南中未明确指定 Python Docstring 的格式标准（如 Google, Sphinx, 或 NumPy）。若团队成员各写各的，后续用工具生成网页文档时格式会完全错乱。 | **强制绑定 NumPy Style**。鉴于 MCPC 深度依赖 `scipy`、`astropy` 等科学计算库，采用同源的 NumPy 风格是最自然的。它对长篇数学公式和多返回值参数的排版支持最好，也完美契合 Sphinx 的 `numpydoc` 扩展。 |
| **文档化粒度**     | 要求所有类和函数都具备详细文档，容易导致“文档疲劳”，让开发者在敏捷迭代时感到痛苦。 | **区分内外部接口**。建议规定：只对 `core/` 目录下的核心抽象类（如 `IForceModel`）和 `mission_sim/__init__.py` 暴露的公共 API 强制要求全量文档。对于内部辅助函数（如以下划线 `_` 开头的方法），坚持**“代码即文档”**（清晰的命名 + 类型注解）即可。 |
| **架构图的维护**   | 推荐使用 UML，但如果依赖外部画图软件（如 Visio），架构图很快就会因为更新不及时而“腐烂”。 | **Doc as Code (文档即代码)**。既然已经在用 PlantUML，建议在指南中强制要求：所有架构图必须以 `.puml` 纯文本格式随同 Python 源码一起提交到 Git 仓库，实现版本同构。 |
